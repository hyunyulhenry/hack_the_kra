## 시장이 틀릴 수 있는 지점 (엣지 후보)

| 편향 | 설명 |
|------|------|
| Favorite-Longshot Bias | 대중이 장거리 배당마를 과대평가, 인기마를 과소평가 |
| Recency Bias | 최근 1~2경주에 과도한 가중치, 장기 폼 변화 무시 |
| 장구/장비 변경 | 블링커 최초 착용, 텅타이 등 — 캐주얼 베터가 무시하는 강한 시그널 |
| 포스트 포지션 편향 | KRA 트랙 구조에 따른 게이트별 유불리를 시장이 완전히 반영 못함 |
| 기수 변경 | 당일 기수 교체는 강한 시그널인데 배당에 늦게 반영됨 |
| 페이스 시나리오 | 선행마가 몰릴 때 추입마 유리 — 대중은 경주 형태를 예측 못함 |
| 중량 효과의 비선형성 | 58kg→60kg의 영향이 52kg→54kg과 다름 |

## 벤터 Logit의 한계와 ML 보완

### Multinomial Logit의 한계
- 피처와 승리확률 간 **선형 관계만** 가정
- 피처 간 **상호작용을 수동으로** 만들어야 함
- "이 말은 1200m에서 강한데, 1200m이면서 비 올 때만 강하다" → 캡처 불가

### LightGBM이 해결하는 것
- 비선형 효과 자동 캡처 (거리 선호도가 특정 거리에서만 작동)
- 피처 상호작용 자동 탐지 (기수 × 트랙 × 거리 조합)
- 결측치 자체 처리 (경마 데이터는 결측이 많음)

### 데이터 규모 고려
- 서울 3년치 약 3,600경주 × 평균 13두 = ~47,000 관측치
- 복잡한 딥러닝은 과적합 위험이 큼
- **LightGBM 수준이 데이터 규모에 가장 적합**

## 3-Stage 아키텍처

```
┌─────────────────────────────────────────┐
│ Feature Selection (전처리)               │
│ SHAP 기반 중요도 → 하위 피처 제거         │
│ 과적합 방지, 노이즈 피처 차단             │
├─────────────────────────────────────────┤
│ Stage 1: Fundamental Model              │
│ Stacking Ensemble (v3~)                 │
│   Base: LightGBM + XGBoost + CatBoost  │
│   Meta: XGBoost                         │
│ Loss: LambdaRank (순위 최적화)           │
├─────────────────────────────────────────┤
│ Stage 2: Combined Model (벤터 핵심)      │
│ raw_prob + public_odds → final_prob     │
│ α를 학습 (고정 아님)                     │
├─────────────────────────────────────────┤
│ Stage 3: Betting Decision               │
│ Fractional Kelly (1/4~1/2 Kelly)        │
│ 승식별 최적화 (단승/복승/쌍승)            │
└─────────────────────────────────────────┘
```

### Stage 1: Fundamental Model
- **LambdaRank 손실함수**: 이진 분류가 아니라 순위 자체를 최적화
- 경마는 1등만 중요한 게 아님 — 2등, 3등도 복승/삼복승에 필요
- 기존 log-loss보다 실전 베팅 수익과 더 직접적으로 연결
- **v3부터 스태킹 앙상블 적용** (Nguyen 2024):
  - Base 모델: LightGBM + XGBoost + **CatBoost**
  - Meta 모델: XGBoost (base 예측값을 입력으로 최종 예측)
  - CatBoost 추가 이유: 기수명/조교사명/등급코드 등 범주형 변수를 원핫 없이 직접 처리

### Stage 2: Combined Model
- 벤터의 최대 발견을 계승
- 모델이 아무리 좋아도 공중 배당률 없이는 성능이 급격히 저하
- `final_logit = α × log(odds) + β × model_score + γ × (model_score × log(odds))`
- 상호작용 항(γ): "시장이 과소평가하고 모델도 높게 볼 때" 시너지 캡처

### Stage 3: Betting Decision
- Fractional Kelly (1/4~1/2) — full Kelly는 변동성이 너무 큼
- 확률 보정(calibration) 필수 → Platt Scaling 또는 Isotonic Regression
- KRA 환수율 반영 (단승 80%, 복승 73%)

## 피처 엔지니어링

| 카테고리 | 피처 | 벤터 有/無 | 임팩트 |
|---------|------|-----------|--------|
| Speed Figure | 최근 N경주 주파 시간, 트랙/거리/주로상태 정규화 | 有 | 높음 |
| Class Rating | 이긴 상대의 레이팅 가중 평균 | 일부 | 높음 |
| Form Cycle | 레이팅 이동평균의 기울기 (상승/하강 추세) | 無 | 높음 |
| Pace Profile | 선행/중단/추입 성향, 당일 예상 페이스 구도 | 無 | 높음 |
| Post × Distance | 특정 거리에서 특정 게이트의 역사적 승률 편향 | 일부 | 중간 |
| 기수-말 궁합 | 이 기수가 이 말을 탔을 때 vs 평균 기수 탈 때 성적 차이 → 불일치 점수화 | 無 | 중간 |
| Trainer Intent | 경주 간격, 등급/거리 변경 → 조교사 의도 추정 | 無 | 중간 |
| Equipment Change | 블링커 최초/추가/제거 → 원핫 인코딩 | 有(API) | 중간 |
| 체중 변화율 | 전회 대비 체중 변화 % | 有(Phase2) | 중간 |

### 가장 임팩트 큰 새 피처 3개
1. **Form Cycle (추세)** — 레이팅이 올라가는 중인 말 vs 내려가는 말
2. **Pace Scenario** — 당일 선행마 수로 경주 형태 예측
3. **기수-말 궁합** — 특정 조합의 역사적 성적

## Feature Selection (피처 선택)

피처 엔지니어링 후 **선택 단계를 명시적으로 분리** (Gupta & Singh 2023).
피처가 많아질수록 47,000 관측치 규모에서 과적합 위험이 높아짐.

```python
import shap

# 1. LightGBM으로 1차 학습
model.fit(X_train, y_train)

# 2. SHAP 중요도 계산
explainer = shap.TreeExplainer(model)
shap_values = explainer.shap_values(X_train)
importance = np.abs(shap_values).mean(axis=0)

# 3. 하위 N% 피처 제거 후 재학습
threshold = np.percentile(importance, 20)  # 하위 20% 제거
selected_features = X_train.columns[importance >= threshold]
```

- SHAP 기반 선택이 단순 feature_importance보다 신뢰성 높음
- 제거 기준: 하위 10~20% (교차검증으로 최적 비율 결정)
- 피처 선택 결과를 문서화 → 어떤 피처가 실제로 중요한지 파악

## 개발 로드맵

| 단계 | 모델 | 목적 |
|------|------|------|
| v1 | Multinomial Logit + 공중 배당률 | 벤터 재현, Δr² 측정, 파이프라인 검증 |
| v2 | LightGBM + 공중 배당률 | 비선형 효과 캡처, v1 대비 Δr² 개선 확인 |
| v3 | LightGBM + XGBoost + CatBoost → Stacking | 순위 최적화, 범주형 처리 개선, 복식 베팅 확장 |
| v4 | Ensemble + 페이스 피처 + 실시간 배당 | Phase 3 데이터 활용, 최종 실전 모델 |

### 핵심 원칙
1. 모델 선택보다 피처 엔지니어링이 더 중요
2. 공중 배당률 통합은 필수 (30년간 변하지 않은 진리)
3. Logit → LightGBM 교체만으로 의미 있는 개선 가능
4. Calibration을 빼먹으면 Kelly가 계좌를 날림
5. v1(벤터 재현)을 건너뛰지 말 것 — 기본이 안 되면 ML도 무의미
